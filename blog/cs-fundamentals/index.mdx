# 背景

- 基礎知識は大切。
- 基礎が分かっていなければ応用が効かない。
- 技術を深めていきたいのであれば基礎は必須である。雰囲気でしか分かっていない知識が多い。
- アプリケーションレイヤーのコードしか書いていないと、普段意識することが少ない。
- 最低限言語化して説明でき、長所短所、ユースケース等を説明できる程度の知識をキープするのは必要だろうと思った。
- 一旦立ち帰って復習したり、そもそも足りていない知識をコツコツ補っていきたい

とりあえず書き出してみた

## 情報基礎
- プログラムはなぜ動くのか
  - メモリにロードされて実行されるまで説明できない
  - データ構造
      - プログラムの計算量、求め方
          - 最低限知っている
      - リストと配列の違い
          - 怪しい
  - メモリ空間・スレッド・プロセス
      - 雰囲気でしか分かっていない。説明できない。
- ネットワーク
  - ブラウザにURLを入力してから画面が表示されるまで
    - 最低限説明できる。順に
      - DNSサーバー問い合わせ
      - TCP 3ウェイハンドシェイクでコネクション確率
      - HTTPSならTLSネゴシエーション
        - 公開鍵暗号と共通鍵暗号のハイブリッド
        - サーバー証明書をCA(認証局)の公開鍵で検証
        - サーバーの公開鍵を使って秘密鍵を生成
        - 以後暗号化通信
      - HTTP
      - ブラウザがHTMLレンダリングとコンテンツフェッチ
## セキュリティ
- そもそもセキュリティはクリティカルな上に求められるレベルが高い上に知識が乏しすぎる。絶対に自分では実装せずに、フレームワークに頼る、FirebaseAuthentication 使うとかが基本だと思う自分においては。
- 暗号化
  - 公開鍵暗号、共通鍵暗号
    - どんなものかの説明はできる。アルゴリズムは知らない。
  - パスワード
    - ソルトとハッシュ関数でストレッチングをしてセキュアに保存するよ程度は知っている
- 攻撃手法
  - CSRF, XSS, SQLインジェクション, セッションハイジャックと超基本的な対策はわかる
    - CSRF
      - トークン
    - XSS, SQLインジェクション
      - エスケープ
    - セッションハイジャック
      - 説明できない

  - 最低限説明できる
- SSR・CSR
  - SSR
    - Pros
      - SEO,初期表示の速度
    - Cons
      - リクエスト毎に新たなHTMLを生成するのでCSRに比べて遅くなる
      - サーバー負荷が増える
  - CSR
    - Pros
      - サーバ負荷軽減
    - Cons
      - 初期ロード
